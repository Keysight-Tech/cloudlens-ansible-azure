One command, full stack

vController, KVO, and vPB plus sensors. End to end. From a single paste.

curl -sSL https://raw.githubusercontent.com/Keysight-Tech/cloudlens-ansible-azure/main/deploy/deploy-stack.sh | bash

Prompts for resource group and region, accepts Marketplace terms, deploys vController, waits for initialization, optionally adds vPB, and chains to sensor deployment. Run from Azure Cloud Shell or any machine with Azure CLI.

Run from Docker

Pinned image with Ansible, Azure SDKs, and pywinrm. Works from your laptop, a CI runner, or any container host.

docker run --rm -it \
  -v $(pwd)/customer_input.yaml:/work/customer_input.yaml \
  -v $HOME/.ssh:/root/.ssh:ro \
  -e AZURE_SUBSCRIPTION_ID -e AZURE_TENANT \
  -e AZURE_CLIENT_ID -e AZURE_SECRET \
  ghcr.io/keysight-tech/cloudlens-ansible-azure:latest

Service Principal credentials passed via env vars (use scripts/setup_azure_sp.sh to create one). Same behavior as the Cloud Shell quickstart, just packaged for CI/CD or repeat customer deployments.

CloudLens vController formerly CLMS

Central management for sensors. Required for any sensor deployment.

  • Version 6.14.0-89 from Azure Marketplace
  • One VM, public IP, ready in 15 minutes
  • Default UI: admin / Cl0udLens@dm!n

Vision Orchestrator (KVO) NEW

Optional. Centralizes vPB fleet configuration and automation across physical, virtual, and cloud.

  • Version 3.0.0-55 from Azure Marketplace
  • One VM, public IP, single NIC control plane
  • Manages multiple vController and vPB fleets
▶ Deploy KVO

Virtual Packet Broker (vPB)

Optional. For inline traffic processing and VXLAN aggregation.

  • Version 3.15.0-1 from Azure Marketplace
  • Three-NIC layout with accelerated networking
  • CLI access via two-hop SSH
▶ Deploy vPB

In the Azure Portal, the offers appear as "Keysight CloudLens vController", "Keysight Vision Orchestrator", and "Keysight CloudLens Virtual Packet Broker". The Deploy buttons above point to the correct offer IDs automatically. CLMS was renamed to vController in 2026.

After vController finishes initializing (about 15 minutes), open the UI, create a project, copy the project key, then return here to run the sensor deployment.

First-time use of these images requires accepting Marketplace terms. The Portal will prompt you on first deploy, or run az vm image terms accept --publisher keysight-technologies-cloudlens --offer <offer> --plan <plan> (use keysight-technologies-kvop for KVO).

Get started in 5 steps

Most deployments under 50 VMs complete in 5 to 10 minutes.

  1. 1
    ☑ Check prerequisites 2 minutes

    You need these in place before deploying:

    • Azure subscription with Reader and Virtual Machine Contributor roles
    • CloudLens vController (formerly CLMS) deployed and reachable from your target VMs
    • A project key from vController (Settings, Projects, API Keys)
    • SSH key (for Linux targets) or admin password (for Windows targets)

    If you do not have vController deployed yet, use the Deploy vController button above.

  2. 2
    🏷 Tag your Azure VMs 5 minutes

    The deployment discovers VMs by tag. Apply these to every target VM:

    TagValue
    cloudlensyes
    osubuntu or rhel or windows
    envprod (or dev, qa)

    Bulk tag a resource group:

    for vm in $(az vm list -g <RG> --query "[?storageProfile.osDisk.osType=='Linux'].name" -o tsv); do
  az vm update -g <RG> -n $vm --set tags.cloudlens=yes tags.os=ubuntu tags.env=prod
done
  3. 3
    ⚙ Build your config file 1 minute

    Use the wizard below to fill four fields and download customer_input.yaml.

    Jump to wizard ↓

    Tip: the wizard shows the YAML rendering live as you type. Each field has inline help so you know exactly what to enter.

  4. 4
    🚀 Deploy 5 to 60 minutes

    Place your downloaded customer_input.yaml in your working directory, then pick one of the three deploy paths.

    🌐 Tier 1: Azure Portal

    One click. No local tools. Best for: trying it for the first time.

    Deploy from Portal

    ☁ Tier 2: Cloud Shell

    Already authenticated. One curl command. Best for: most customers.

    curl -sSL https://raw.githubusercontent.com/Keysight-Tech/cloudlens-ansible-azure/main/quickstart.sh | bash

    🐋 Tier 3: Docker

    Reproducible. Pinned versions. Best for: CI or repeat deployments.

    docker run --rm -it -v $(pwd)/customer_input.yaml:/work/customer_input.yaml ghcr.io/keysight-tech/cloudlens-ansible-azure:latest
  5. 5
    ✅ Verify in vController 2 minutes

    Open your vController web UI and confirm:

    • Each tagged VM appears as a registered sensor (under Sensors)
    • Custom tags from your config match what you see in vController
    • Sensors show as connected and reporting

    Stuck? See the troubleshooting section below or the full troubleshooting docs.

Build your customer_input.yaml

Fill in four fields. Download the file. Run quickstart. That is the whole flow.

1

CloudLens vController

Where do I find this?

The IP or hostname of your vController deployment. You can find it in your Azure Portal under the vController VM details, or in the output of your vController Terraform deployment.

2

Project key

Where do I find this?

In vController web UI: Settings, Projects, API Keys, copy the key for the project you want sensors to register under.

3

Custom tags

What does this do?

These tags are attached to each registered sensor and visible in vController. Use them to filter and group sensors (for example: Customer=Acme, Region=eastus2).

4

Resource groups

What if I leave this blank?

Blank means scan ALL resource groups in your subscription for tagged VMs. List specific RGs (comma-separated) to limit discovery.

Common issues

If something does not work, check these first.

I don't have vController (formerly CLMS) yet. Where do I start?

Use the Deploy vController button at the top of this page. vController comes from Azure Marketplace, deploys in 15 minutes, and is ready before you start tagging your VMs. While you are there, also deploy KVO if you want centralized fleet orchestration across multiple vPB instances.

vPB just deployed. Why can't I SSH yet?

The vPB image uses port 9022 for SSH, NOT the default port 22. Keysight CloudLens OS (KCOS) binds sshd only to 9022 on the public interface. Connections to :22 will time out forever even after the VM is fully booted.

What to do: wait 10-15 minutes after deploy succeeds for KCOS to initialize, then:

ssh azureuser@<vpb-public-ip> -p 9022          # OS shell
ssh admin@localhost -p 2222                       # vPB CLI (admin / ixia)

NSG rule: make sure your Network Security Group on the vPB management NIC allows inbound TCP on port 9022. The marketplace ARM template opens this automatically; if you deployed via a custom path, add the rule manually.

Same vController initialization note: the web UI on port 443 is usually ready within 60 seconds, but full system initialization (including the registry and project creation) takes around 15 minutes.

Inventory finds 0 VMs

Cause: tags missing or misspelled.

Fix: verify with az vm show -g RG -n VM --query tags. The cloudlens, os, and env tags must all be present.

SSH Permission denied

Cause: your public key is not on the target VM.

Fix: bootstrap with az vm run-command invoke to push your key, then re-run the deploy.

WinRM timeout on Windows VMs

Cause: WinRM disabled by default on Windows Server.

Fix: run playbooks/bootstrap_windows_winrm.yaml first to enable WinRM via Azure Run Command.

Sensor does not appear in vController

Cause: wrong project key, or sensor cannot reach vController network.

Fix: double-check the project key from vController UI. Verify the VM can reach vController on port 443.

How long should this take?

1 to 50 VMs: 5 to 10 minutes.

50 to 500 VMs: 15 to 30 minutes.

500 to 2000 VMs: 30 to 60 minutes.

Thousands: sharded automatically, 30 to 60 minutes for up to 10K.
