One command, full stack

CLMS, vPB, and sensors. End to end. From a single paste.

curl -sSL https://raw.githubusercontent.com/Keysight-Tech/cloudlens-ansible-azure/main/deploy/deploy-stack.sh | bash

Prompts for resource group and region, accepts Marketplace terms, deploys CLMS, waits for initialization, optionally adds vPB, and chains to sensor deployment. Run from Azure Cloud Shell or any machine with Azure CLI.

Don't have CLMS or vPB yet?

Deploy them from Azure Marketplace in one click before running sensor deployments.

CloudLens Manager (CLMS)

Central management for sensors. Required for any sensor deployment.

  • Version 6.13.076 from Azure Marketplace
  • One VM, public IP, ready in 15 minutes
  • Default UI: admin / Cl0udLens@dm!n
▶ Deploy CLMS

Virtual Packet Broker (vPB)

Optional. For inline traffic processing and VXLAN aggregation.

  • Version 3.15.01 from Azure Marketplace
  • Three-NIC layout with accelerated networking
  • CLI access via two-hop SSH
▶ Deploy vPB

In the Azure Portal, the CLMS offer appears as "CloudLens Manager (Preview)" and the vPB offer appears as "CloudLens Virtual Packet Broker". The Deploy buttons above point to the correct offer IDs automatically.

After CLMS finishes initializing (about 15 minutes), open the UI, create a project, copy the project key, then return here to run the sensor deployment.

First-time use of these images requires accepting Marketplace terms. The Portal will prompt you on first deploy, or run az vm image terms accept --publisher keysight-technologies-cloudlens --offer <offer> --plan <plan>.

Prefer Terraform?

Both CLMS and vPB are also available as Terraform modules in deploy/terraform/. Same marketplace images, IaC-friendly for pipelines and repeatable deployments.

Get started in 5 steps

Most deployments under 50 VMs complete in 5 to 10 minutes.

  1. 1
    ☑ Check prerequisites 2 minutes

    You need these in place before deploying:

    • Azure subscription with Reader and Virtual Machine Contributor roles
    • CloudLens Manager (CLMS) deployed and reachable from your target VMs
    • A project key from CLMS (Settings, Projects, API Keys)
    • SSH key (for Linux targets) or admin password (for Windows targets)

    If you do not have CLMS deployed yet, use the Deploy CLMS button above.

  2. 2
    🏷 Tag your Azure VMs 5 minutes

    The deployment discovers VMs by tag. Apply these to every target VM:

    TagValue
    cloudlensyes
    osubuntu or rhel or windows
    envprod (or dev, qa)

    Bulk tag a resource group:

    for vm in $(az vm list -g <RG> --query "[?storageProfile.osDisk.osType=='Linux'].name" -o tsv); do
  az vm update -g <RG> -n $vm --set tags.cloudlens=yes tags.os=ubuntu tags.env=prod
done
  3. 3
    ⚙ Build your config file 1 minute

    Use the wizard below to fill four fields and download customer_input.yaml.

    Jump to wizard ↓

    Tip: the wizard shows the YAML rendering live as you type. Each field has inline help so you know exactly what to enter.

  4. 4
    🚀 Deploy 5 to 60 minutes

    Place your downloaded customer_input.yaml in your working directory, then pick one of the three deploy paths.

    🌐 Tier 1: Azure Portal

    One click. No local tools. Best for: trying it for the first time.

    Deploy from Portal

    ☁ Tier 2: Cloud Shell

    Already authenticated. One curl command. Best for: most customers.

    curl -sSL https://raw.githubusercontent.com/Keysight-Tech/cloudlens-ansible-azure/main/quickstart.sh | bash

    🐋 Tier 3: Docker

    Reproducible. Pinned versions. Best for: CI or repeat deployments.

    docker run --rm -it -v $(pwd)/customer_input.yaml:/work/customer_input.yaml ghcr.io/keysight-tech/cloudlens-ansible-azure:latest
  5. 5
    ✅ Verify in CLMS 2 minutes

    Open your CLMS web UI and confirm:

    • Each tagged VM appears as a registered sensor (under Sensors)
    • Custom tags from your config match what you see in CLMS
    • Sensors show as connected and reporting

    Stuck? See the troubleshooting section below or the full troubleshooting docs.

Build your customer_input.yaml

Fill in four fields. Download the file. Run quickstart. That is the whole flow.

1

CloudLens Manager

Where do I find this?

The IP or hostname of your CloudLens Manager deployment. You can find it in your Azure Portal under the CLMS VM details, or in the output of your CLMS Terraform deployment.

2

Project key

Where do I find this?

In CLMS web UI: Settings, Projects, API Keys, copy the key for the project you want sensors to register under.

3

Custom tags

What does this do?

These tags are attached to each registered sensor and visible in CLMS. Use them to filter and group sensors (for example: Customer=Acme, Region=eastus2).

4

Resource groups

What if I leave this blank?

Blank means scan ALL resource groups in your subscription for tagged VMs. List specific RGs (comma-separated) to limit discovery.

Once you have customer_input.yaml

  1. Save the file to your working directory
  2. Choose a deploy path above (Portal, Cloud Shell, or Docker)
  3. The deploy command will pick up your customer_input.yaml automatically

Common issues

If something does not work, check these first.

I don't have CLMS yet. Where do I start?

Use the Deploy CLMS button at the top of this page. CLMS comes from Azure Marketplace, deploys in 15 minutes, and is ready before you start tagging your VMs.

vPB just deployed. Why can't I SSH yet?

Expected: vPB takes 10 to 15 minutes after the Azure deployment finishes before SSH on port 22 is reachable. The VM itself is up, but the internal CLI service and SSH daemon are still initializing.

What to do: wait 15 minutes after deploy succeeds, then try ssh azureuser@<vpb-public-ip>. From there, two-hop into the CLI with ssh admin@localhost -p 2222 (default password: ixia).

Same applies to CLMS: the web UI on port 443 is usually ready within 60 seconds, but full system initialization (including the registry and project creation) takes around 15 minutes.

Inventory finds 0 VMs

Cause: tags missing or misspelled.

Fix: verify with az vm show -g RG -n VM --query tags. The cloudlens, os, and env tags must all be present.

SSH Permission denied

Cause: your public key is not on the target VM.

Fix: bootstrap with az vm run-command invoke to push your key, then re-run the deploy.

WinRM timeout on Windows VMs

Cause: WinRM disabled by default on Windows Server.

Fix: run playbooks/bootstrap_windows_winrm.yaml first to enable WinRM via Azure Run Command.

Sensor does not appear in CLMS

Cause: wrong project key, or sensor cannot reach CLMS network.

Fix: double-check the project key from CLMS UI. Verify the VM can reach CLMS on port 443.

How long should this take?

1 to 50 VMs: 5 to 10 minutes.

50 to 500 VMs: 15 to 30 minutes.

500 to 2000 VMs: 30 to 60 minutes.

Thousands: sharded automatically, 30 to 60 minutes for up to 10K.
